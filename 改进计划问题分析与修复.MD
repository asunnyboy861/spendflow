3. **影响代码查找和维护**：
   - 开发者习惯在 `Core/Domain/Models/` 查找模型

## 二、架构模式不一致问题

### 2.1 问题描述

**现有代码的架构模式**：
```swift
// ✅ 正确：通过Repository操作数据
class DashboardViewModel: ObservableObject {
    private let transactionRepository: TransactionRepository
    private let budgetRepository: BudgetRepository
    
    init(
        transactionRepository: TransactionRepository,
        budgetRepository: BudgetRepository
    ) {
        self.transactionRepository = transactionRepository
        self.budgetRepository = budgetRepository
    }
    
    func loadData() {
        let transactions = transactionRepository.fetchAll()
        // ...
    }
}
```

**改进计划中的错误示例**：
```swift
// ❌ 错误：直接操作CoreData
class TransactionImporter {
    func importTransactions(_ plaidTransactions: [PlaidTransaction]) {
        for plaidTransaction in plaidTransactions {
            let transaction = Transaction(...)
            // 直接保存到CoreData
            try context.save()
        }
    }
}
```

### 2.2 问题分析

1. **违反Repository模式**：
   - 现有代码所有数据操作都通过Repository
   - 改进计划中的服务直接操作数据，绕过了Repository层

2. **破坏分层架构**：
   - Service层应该调用Repository，而不是直接操作数据
   - 这样可以保持数据访问的一致性和可测试性

3. **影响依赖注入**：
   - 现有代码使用依赖注入Repository
   - 直接操作数据破坏了依赖注入模式

### 2.3 修复方案

**正确的实现方式**：

```swift
// ✅ 正确：通过Repository操作数据
class TransactionImporter {
    private let transactionRepository: TransactionRepository
    private let categorySuggester: CategorySuggester
    
    init(
        transactionRepository: TransactionRepository,
        categorySuggester: CategorySuggester
    ) {
        self.transactionRepository = transactionRepository
        self.categorySuggester = categorySuggester
    }
    
    func importTransactions(_ plaidTransactions: [PlaidTransaction]) async throws {
        for plaidTransaction in plaidTransactions {
            let suggestedCategory = categorySuggester.suggestCategory(
                for: plaidTransaction.name,
                amount: plaidTransaction.amount
            )
            
            let transaction = Transaction(
                amount: abs(plaidTransaction.amount),
                category: suggestedCategory.name,
                date: plaidTransaction.date,
                note: plaidTransaction.merchantName ?? plaidTransaction.name,
                type: plaidTransaction.amount < 0 ? .income : .expense
            )
            
            // ✅ 通过Repository保存
            try transactionRepository.save(transaction)
        }
    }
}
```

### 4.2 Plaid集成问题

**问题描述**：
- 改进计划中提到使用Plaid API
- 但Plaid需要后端服务来交换access token
- 文档中没有详细说明后端架构

**修复方案**：
1. **添加后端架构说明**：

   - 需要处理OAuth回调
   - 需要安全存储access token

3. **提供替代方案**：
   - 初期可以使用手动导入CSV

### 4.3 通知权限问题

**问题描述**：
- 改进计划中提到推送通知
- 但没有提到需要申请通知权限
- 没有处理用户拒绝权限的情况

**修复方案**：
```swift
// ✅ 添加权限请求
class NotificationService {
    func requestPermission() async throws -> Bool {
        let center = UNUserNotificationCenter.current()
        let options: UNAuthorizationOptions = [.alert, .sound, .badge]
        let granted = try await center.requestAuthorization(options: options)
        
        if !granted {
            throw NotificationError.permissionDenied
        }
        
        return granted
    }
}

// ✅ 在ViewModel中处理权限
class BillsViewModel: ObservableObject {
    func addBill(_ bill: Bill) {
        Task {
            do {
                // 先请求权限
                try await notificationService.requestPermission()
                // 然后添加账单
                try billRepository.save(bill)
                try await billReminderService.scheduleReminder(for: bill)
            } catch NotificationError.permissionDenied {
                // 处理权限被拒绝
                await MainActor.run {
                    self.showPermissionAlert = true
                }
            }
        }
    }
}
```

## 七、修复后的改进计划要点

### 7.1 文件结构规范

**强制要求**：
1. 所有Models放在 `Core/Domain/Models/`
2. 所有ViewModels放在 `Core/Presentation/ViewModels/`
3. 所有Repository协议放在 `Core/Domain/Repositories/`
4. 所有Repository实现放在 `Core/Data/Repositories/`
5. Features模块只包含Views、Components、Services

### 7.2 架构模式规范

**强制要求**：
1. 所有数据操作必须通过Repository
2. Service层只能调用Repository，不能直接操作数据
3. 使用依赖注入，不能硬编码依赖
4. ViewModel通过Repository获取数据

### 7.3 代码复用规范

**强制要求**：
1. 新UI必须使用 `cardStyle()`
2. 新按钮必须使用 `HapticButton`
3. 新空状态必须使用 `EmptyStateView`
4. 所有间距使用 `DesignTokens.Spacing`
5. 所有颜色使用 `Color+Theme` 扩展

### 7.4 技术实现规范

**强制要求**：
1. 明确iOS版本要求（iOS 15+ 或 iOS 16+）
2. 提供降级方案（针对iOS 15）
3. 明确后端服务要求
4. 明确第三方服务要求（Plaid等）
5. 处理权限请求和拒绝情况

### 7.5 命名规范

**强制要求**：
1. ViewModel命名：`XXXViewModel`
2. Service协议命名：`XXXService`
3. Service实现命名：`DefaultXXXService` 或 `XXXServiceImpl`
4. Repository协议命名：`XXXRepository`
5. Repository实现命名：`XXXRepositoryImpl`

### 7.6 集成规范

**强制要求**：
1. 新功能必须说明如何与现有功能集成
2. 新数据类型必须说明如何与现有数据模型集成
3. 新服务必须说明如何与现有服务集成
4. 提供集成测试用例
